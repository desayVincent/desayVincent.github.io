c++ 内存排序

发表于 2024-10-16 更新于 2024-10-17 分类于 C/C++ 阅读次数：
为了解决内存排序，C++引入了内存顺序选项（如std::memory_order_relaxed、std::memory_order_acquire、std::memory_order_release等）。这些选项允许程序员精确控制内存操作的顺序，在保证程序正确性的同时，还能够在某些情况下提高性能。

产生内存顺序的原因

  1. 处理器乱序执行：

    现代CPU为了提高性能，会对指令进行乱序执行。例如，可能会先执行后面的内存读取，而不是按程序顺序执行。这在单线程中通常不会有问题，但在多线程环境下可能导致意外行为。

  2. 编译器优化：

    编译器可能会重排指令顺序以提高性能。例如，可能会将看似无关的操作重新排序，这在多线程环境中可能导致问题。

  3. 缓存一致性：

    在多核系统中，每个核心都有自己的缓存。当一个核心修改了内存中的值，其他核心可能不会立即看到这个修改，因为它们可能仍在使用自己缓存中的旧值。

  4. 存储缓冲区：

    处理器可能会将写操作暂存在存储缓冲区中，而不是立即写入主内存。这可能导致其他处理器看到写操作的顺序与程序中的顺序不同。

  5. 多级内存架构：

    现代计算机使用多级内存架构（如L1、L2、L3缓存和主内存），不同级别的内存访问速度不同，这可能导致内存操作的可见性问题。

内存顺序的类型

1.memory_order_relaxed

  • 最宽松的内存序，只保证原子性，不提供同步或顺序保证。
  • 适用于计数器等简单场景，性能最好。

2.memory_order_acquire

  • 用于读操作，保证当前读取之后的内存访问不会被重排到这个操作之前。
  • 通常与memory_order_release配对使用。

3.memory_order_release

  • 用于写操作，保证当前写入之前的内存访问不会被重排到这个操作之后。
  • 通常与memory_order_acquire配对使用。

4.memory_order_acq_rel

  • 结合了acquire和release语义，用于读-修改-写操作。

5.memory_order_seq_cst

  • 最严格的内存序，提供全局的顺序一致性。
  • 性能开销最大，但最容易理解和使用。

6.memory_order_consume

  • 类似于acquire，但只对依赖于加载值的操作提供顺序保证。
  • 在实践中较少使用，因为实现复杂且效果不明显。

Sequentially-consistent ordering

顺序一致性是最强的排序类型,也是默认的排序类型。它保证在设置原子变量后,该变量立即对所有其他线程可见

Release-Acquire ordering

释放-获取排序必须成对使用。memory_order_release -> memory_order_acquire,它保证原子变量对获取该变量的线程可见

Relaxed ordering

松散排序是最弱的排序类型,这意味着原子变量最终会对其他线程可见。但没有时间(顺序)保证。

这三种排序类型代表了C++内存模型中不同级别的同步保证:

  • 顺序一致性提供了最强的保证,但可能带来最大的性能开销。
  • 释放-获取排序提供了一种在特定线程间建立同步点的方法,通常用于生产者-消费者模式。
  • 松散排序提供了最小的保证,但允许最大的优化空间,适用于简单的计数器等场景。

例子

memory_order_relaxed

memory_order_relaxed 是最宽松的内存序，它只保证原子性，不提供任何同步或顺序保证。这种内存序通常用于以下场景：

  1. 计数器
  2. 统计信息收集
  3. 某些无锁数据结构

在这个例子中：

  1. 我们有一个全局的原子计数器 global_counter。

  2. 多个线程同时递增这个计数器，每个线程递增 100000 次。

  3. 我们使用 memory_order_relaxed 进行递增操作，因为：

    • 我们只关心最终的计数结果，不关心中间过程。

    • 我们不需要这个计数器操作与其他内存操作同步。

    • 我们可以容忍在程序的其他部分暂时看到旧值。

  4. 使用 memory_order_relaxed 可能会比使用默认的 memory_order_seq_cst 有更好的性能，特别是在一些架构上。

这个例子展示了 memory_order_relaxed 的典型用途：当我们只需要保证操作的原子性，而不需要建立线程间的同步点时，可以使用这种内存序。这在一些高性能场景下特别有用，比如计数器、统计信息收集等，在这些场景中，我们通常只关心最终结果，而不关心中间状态。

需要注意的是，虽然 memory_order_relaxed 可以提供更好的性能，但它也是最难正确使用的内存序。在使用时需要非常小心，确保程序的正确性不依赖于任何同步或顺序保证。

主要的解释:

  1. 在 push 函数中:

    • 我们使用 std::memory_order_acquire 来加载 head。这确保了如果队列不满,我们能看到最新的 head 值。

    • 我们使用 std::memory_order_release 来存储新的 tail 值。这确保了所有之前的写操作(包括向 buffer 写入数据)在更新 tail 之前完成。

  2. 在 pop 函数中:

    • 我们使用 std::memory_order_acquire 来加载 tail。这确保了如果队列不空,我们能看到最新的 tail 值,以及所有在 push 操作中发生的写入。

    • 我们使用 std::memory_order_release 来存储新的 head 值。这确保了所有之前的读操作(包括从 buffer 读取数据)在更新 head 之前完成。

  3. 我们仍然对 head 和 tail 的初始加载使用 std::memory_order_relaxed,因为这些操作不需要同步。

这种实现方式建立了正确的 release-acquire 同步对:

  • 生产者的 release 操作与消费者的 acquire 操作配对,确保消费者能看到生产者写入的数据。
  • 消费者的 release 操作与生产者的 acquire 操作配对,确保生产者不会覆盖尚未被消费的数据。

memory_order_acquire & memory_order_release

用于非原子变量

memory_order_release 的作用不仅仅局限于保证单个变量的内存一致性。它的影响范围更广，涉及到了该操作之前的所有内存写入。

memory_order_release 的保证：

1.写入顺序：

  • 保证 A 的写入在 B 之前完成。

  • 所有在 B 之前的写入操作都不会被重排到 B 之后。

2.可见性：

  • 当其他线程通过 memory_order_acquire 观察到 B 的结果时，A 的结果也必定可见。

3.同步关系：

  • B 和 C 之间建立了同步关系。
  • 这意味着 Thread 1 中 B 之前的所有写入对 Thread 2 中 C 之后的所有读取都是可见的。

memory_order_release 不仅限于 ready.store 操作中使用，但它确实主要用于存储（store）操作：

1.memory_order_release 的主要用途：

  • 它主要用于原子存储操作，如 store()、exchange()、fetch_add() 等。

  • 在示例中，ready.store(true, std::memory_order_release) 是一个典型用法。

2.为什么主要用于存储操作：

  • memory_order_release 建立了一个”释放”语义，确保在这个操作之前的所有内存写入都对其他线程可见。

  • 它通常与 memory_order_acquire（用于加载操作）配对使用，形成同步点。

3.其他可能的用途：

  • memory_order_release 也可以用于其他类型的原子操作，如 atomic_flag::clear()。
  • 在某些复杂的同步模式中，它可能用于非存储操作，但这种用法较少见且需要谨慎。

4.不正确的用法：

  • 不应在加载（load）操作中使用 memory_order_release，这是一个常见的误解。

  • 例如，ready.load(std::memory_order_release) 是不正确的用法。

5.正确的配对：

  • memory_order_release 通常与 memory_order_acquire 配对使用。
  • 在生产者-消费者模式中，生产者使用 release，消费者使用 acquire。
用于memory_order_relaxed

1.使用 memory_order_relaxed 的原子操作：

  • var1 的存储使用 memory_order_relaxed，这是最弱的内存序。
  • memory_order_relaxed 不参与 release-acquire 语义的同步。
  • 编译器和处理器可能会重排这两个操作，因为它们之间没有依赖关系。

2.使用非原子变量：

  • non_atomic_var 是一个普通的非原子变量。
  • C++ 内存模型保证所有非原子操作都会参与 release-acquire 同步。
  • memory_order_release 保证在这个点之前的所有写操作（包括非原子操作）对其他线程可见。

关键点解析：

1.原子操作的独立性：

  • 原子操作（特别是 relaxed 操作）被设计为可以独立优化和重排。
  • memory_order_relaxed 不提供任何跨线程的同步保证，只保证单个变量的原子性。

2.非原子操作的保守处理：

  • 编译器和运行时必须保守地处理非原子操作，因为它们可能导致数据竞争。

  • 因此，非原子写入自动参与到 release-acquire 同步中。

3.Release 语义的范围：

  • memory_order_release 确保其前面的所有写操作（原子和非原子）对获取操作可见。

  • 但这不包括使用 memory_order_relaxed 的独立原子操作。

4.同步与可见性：

  • 在非原子变量的情况下，var2 的 release 操作建立了一个同步点，保证 non_atomic_var 的写入对其他线程可见。

  • 而对于 relaxed 原子操作，这个同步不会扩展到 var1。

5.编译器和硬件优化：

  • 编译器和硬件可能会更积极地优化和重排 relaxed 原子操作。
  • 非原子操作由于潜在的数据竞争风险，会受到更严格的限制。

6.总结：

  • 使用 memory_order_relaxed 的原子操作不参与 release-acquire 同步，因此不能保证其可见性。
  • 非原子操作自动参与 release-acquire 同步，因此可以保证其可见性。

memory_order_acq_rel

1.同步效果：

  • producer 函数使用 memory_order_release 来存储数据。
  • consumer_and_modifier 函数首先用 memory_order_acquire 来检查 ready 标志。
  • 然后使用 memory_order_acq_rel 进行读-修改-写操作。
  • 最后，final_reader 使用 memory_order_acquire 来读取最终结果。

2.保证的顺序：

  • producer 的写入操作 happens-before consumer_and_modifier 的读取。
  • consumer_and_modifier 的修改 happens-before final_reader 的读取。
  • 这样保证了整个操作序列的正确性和可见性。

3.memory_order_acq_rel 的优势：

  • 它在单个原子操作中提供了双向同步。
  • 适用于需要看到之前的写入并确保自己的写入对后续操作可见的场景。

在这个例子中，memory_order_acq_rel 确保了 consumer_and_modifier 能看到 producer 的写入，同时也保证其自身的修改对 final_reader 可见。这种内存序对于实现类似中间处理步骤的操作特别有用，既需要获取之前的数据，又需要发布新的数据。

memory_order_seq_cst

顺序一致性排序是最容易理解的。基本上，它忽略了CPU缓存。一旦原子变量被设置，它就在全局可见并具有全局顺序。这种方式确实会对性能产生一些影响，但除非是对性能特别敏感的地方，否则仍应使用它。

c++默认的内存排序

1.对于原子类型（std::atomic）：

  • 是的，C++ 标准规定，对于所有原子类型的操作，如果没有明确指定内存序，默认使用的是 memory_order_seq_cst。这包括 load、store、exchange、compare_exchange_weak、compare_exchange_strong 等操作。

2.对于非原子操作：

  • 非原子操作不涉及内存序的概念。它们的行为由 C++ 内存模型的其他方面控制，如 happens-before 关系和数据竞争规则。

3.对于其他同步原语：

  • std::mutex、std::condition_variable 等同步原语内部实现了必要的同步，用户通常不需要直接处理内存序。
  • 这些同步原语提供了类似于 memory_order_seq_cst 的保证，确保了操作的可见性和顺序。

这个例子展示了：

  • 原子变量的 load 和 store 操作默认使用 memory_order_seq_cst。
  • std::atomic_flag 的 clear 默认使用 memory_order_seq_cst，而 test_and_set 默认使用 memory_order_acq_rel。
  • 非原子操作不涉及内存序。
  • 互斥锁等同步原语内部处理同步，用户不需要指定内存序。
